> 2024-12 tistory 회고를 5~6년차 시각으로 다시 손본 글입니다. 그때는 final 이 문제구나 → Bean 으로 등록하자 라는 디버깅 흐름을 탔습니다. 지금 다시 들여다보면 시간은 인스턴스가 들고 있는 상태가 아니라 메서드 호출의 결과여야 한다 한 줄로 정리되는 사건이었습니다.

활동 스트림을 최근 3일로 요약해 보여주는 탭에서, 도커 컨테이너가 시동되는 시점의 시간이 영원히 반환되던 버그였습니다. 단순히 시간이 어긋난 정도가 아니라 나노초까지 똑같았습니다. 그 디테일이 처음부터 신호였는데, 그걸 알아보는 데까지 한참이 걸렸습니다.

## 컨테이너 시간만 의심한 곁가지

첫 의심은 컨테이너 시간이었습니다. 호스트에서 `date` 를 찍으면 KST 로 잘 나오는데 컨테이너 안에서는 UTC 로 찍히고 있었으니, 그 어긋남을 먼저 잡아야 한다는 결로 손이 갔습니다. 처음 적용해 둔 `/etc/localtime` 마운트만으로는 충분하지 않았고, 검색해 보니 `/etc/timezone` 으로 맞추는 결을 권장하는 자료가 더 많아 그쪽으로 옮겼습니다. 그래도 여전히 UTC 로 출력이 됐습니다.

결국 Dockerfile 에서 `tzdata` 를 설치하고 zoneinfo 를 직접 복사하는 식으로 한 번 더 다잡았습니다.

```dockerfile
ENV TZ=Asia/Seoul
RUN apk add --no-cache tzdata && \
    cp /usr/share/zoneinfo/$TZ /etc/localtime && \
    echo $TZ > /etc/timezone
```

이렇게 굳히고 나서야 컨테이너 시간은 호스트와 같이 흘러가기 시작했습니다. 다만 API 응답 안의 시간은 여전히 고정이었습니다. `LocalDateTime.now()` 가 매번 같은 값을 줄 리는 없었습니다. 그 자리에서 한 번 더 들여다보지 않으면 디버깅이 진짜 자리로 들어갈 수 없었습니다.

## 진짜 원인 — Util 클래스의 final 필드

문득 머리를 스친 자리는 프로젝트 전체에서 쓰던 `DateUtils` 였습니다. 시간 관련 정의를 한 곳에 모아두려고 작성한 유틸이었는데, 그 안의 시간 변수가 `final` 로 선언되어 있었습니다. 클래스 로딩 시점에 계산된 값을 그대로 들고 있던 셈이고, 컨테이너가 시동되던 그 한 순간의 시간이 모든 API 응답에 박혀 있었던 것입니다. 나노초까지 똑같았던 이유가 거기에 있었습니다.

자책하면서 `final` 을 떼고 `@Component` 를 붙여 Bean 에 등록했습니다. 시간이 매번 새로 계산되리라 기대했지만 결과는 똑같았습니다. 싱글톤 인스턴스의 필드가 또 같은 값을 들고 있었기 때문입니다. 도구를 바꾸기만 했지 발상이 그대로였던 자리고, 한 줄을 손볼 때마다 같은 결과가 나오는 게 더 무서운 자리이기도 했습니다.

| 시도 | 동작 | 왜 실패했는가 |
|---|---|---|
| `static final` 으로 한 번 계산 | 컨테이너 시동 시각 고정 | 시간이 상수가 됨 |
| Bean 등록 + 인스턴스 필드 | 컨테이너 시동 시각 고정 | 싱글톤 + 인스턴스 필드 = 사실상 상수 |
| 일반 클래스 + 사용 지점에서 호출 | 매번 새 값 | 시간을 메서드 호출의 결과로 받음 |

결국 `@Component` 를 떼고 일반 클래스로 되돌린 뒤, 시간이 필요한 자리에서 매번 새로 계산하도록 바꾸고 나서야 정상으로 돌아왔습니다. 테스트를 다 돌리고 개발 서버에 올려 동작을 확인했을 때의 안도감이 지금도 손에 남아 있습니다.

## 한 번 더 들여다보면 — 결의 일반화

이 사건을 한 줄로 환원하면 결이 단순합니다. 시간이라는 값은 "지금 이 호출 시점의 결과" 인데, 인스턴스가 그것을 미리 들고 있으려고 한 자리에서 버그가 시작됐습니다. `final` 도 Bean 등록도 같은 발상의 변형이었습니다. 변하지 않을 값처럼 들고 있다 는 결을, 변할 수밖에 없는 값에 적용한 자리입니다.

같은 결이 시간 하나에만 있는 것은 아닙니다. 매 호출마다 새로 계산되어야 할 값들이 같은 결을 공유합니다.

- `LocalDateTime.now()`, `Instant.now()` 같은 현재 시각
- `Random`, `UUID.randomUUID()` 같은 난수
- 환경 변수의 동적 읽기, feature flag 의 현재 값
- 외부 API 호출이나 DB 조회의 결과

이 값들에 공통점이 하나 있습니다. 호출 시점이라는 컨텍스트가 결과의 일부 입니다. 그러니 인스턴스 필드에 박아두지 않고 메서드 호출의 결과로 받아야 합니다. 사용 지점에서 매번 호출하거나, 함수의 인자로 받아들이거나, 표준의 추상화를 빌리는 결로 갑니다.

## 더 나은 자리 — Clock 추상화

5~6년차 시각에서 다시 같은 코드를 짠다면, 시간을 가져오는 자리에 `java.time.Clock` 을 한 번 끼워두는 결을 우선 고려합니다. 직접 `LocalDateTime.now()` 를 호출하는 대신 주입된 `Clock` 의 `instant()` 로부터 받아 쓰는 형태입니다.

```kotlin
@Service
class ActivityStreamService(
    private val clock: Clock,
) {
    fun summarizeRecent(userId: Long): ActivitySummary {
        val now = LocalDateTime.now(clock)
        val threeDaysAgo = now.minusDays(3)
        // ...
    }
}

@Configuration
class ClockConfig {
    @Bean
    fun clock(): Clock = Clock.systemDefaultZone()
}
```

이 패턴이 좋은 자리는 두 갈래입니다. 첫째, 시간을 인스턴스 필드에 들고 있는 식의 사고가 처음부터 들어설 자리가 사라집니다. 시간은 `Clock` 을 호출한 결과로만 들어오니까요. 둘째, 테스트에서 `Clock.fixed()` 로 시간을 고정할 수 있어 시점 의존 로직의 테스트 가능성이 같이 올라갑니다. 운영 코드와 테스트 코드가 같은 추상화를 통과한다는 점이 그 결의 핵심입니다.

같은 결은 `Supplier<T>` 주입에서도 보입니다. 매 호출마다 새로 계산되어야 할 값이 있다면, 그 값 자체를 주입하지 말고 그 값을 만들어내는 함수 를 주입합니다. 그러면 호출 시점이 결과의 일부 라는 결이 코드에 그대로 박힙니다.

## 정리

테스트에서는 정상인데 운영에서만 나노초까지 똑같이 고정되어 나온다면, 그 값은 메서드 호출의 결과가 아니라 인스턴스가 들고 있는 상태라는 신호 입니다. `final` 이든 Bean 의 인스턴스 필드든, 도구를 바꿔도 발상이 같으면 같은 결과가 나옵니다. 시간·랜덤·현재성을 가진 값은 상태가 아니라 함수 호출의 결과여야 한다 는 한 줄이 그 자리에서 남았고, 이후로는 `Clock` 같은 추상화를 한 겹 끼워두는 결로 코드를 시작합니다.
