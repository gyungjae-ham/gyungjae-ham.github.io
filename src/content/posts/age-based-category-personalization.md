> **TL;DR.** 카테고리 기본 리스팅을 연령대로 개인화하면서, 검색 엔진의 정렬 로직은 한 줄도 새로 짜지 않았습니다. 연령대별 점수는 색인 시점에 상품 문서 안에 미리 구워져 있었고, 쿼리가 할 일은 "이 요청이 어느 점수 필드를 읽을지" 고르는 것뿐이었습니다. 로그인 유저의 연령대가 있으면 그 연령대 점수 필드를, 없으면 전 연령 필드를 읽게 했습니다. 비로그인 경로는 처음부터 영향권 밖이었습니다.

대형 패션 커머스의 카테고리 지면을 연령대로 개인화한 작업의 회고입니다. 백엔드 1인 단독 설계·구현이었고, 잘 풀린 결정과 늦었던 자리를 같이 적습니다.

## 배경

- **대상** 카테고리(예: 상의 > 긴소매) 기본 리스팅. 유저가 살 마음을 먹고 들어오는, 구매 의도가 높은 탐색 경로입니다.
- **규모** 탐색 트래픽이 큰 지면이라, 잘못 건드리면 회귀 피해가 넓습니다.
- **역할** 백엔드 1인 단독 설계·구현.
- **선행 상태** 카테고리 리스팅은 이미 검색 엔진(OpenSearch)의 스코어링으로 순서가 매겨지고 있었습니다. 20대가 보는 긴소매 첫 화면과 50대가 보는 첫 화면이 같은 점수 체계로 정렬됐습니다.

패션은 연령대별 선호가 또렷한 도메인입니다. 같은 카테고리라도 연령대에 맞는 상품을 앞에 두면 첫 화면의 연관도를 올릴 수 있다고 봤습니다. 문제는 **어디를 건드려서** 이걸 하느냐였습니다.

## "연령대 점수로 다시 정렬한다"는 첫 그림은 틀렸다

처음 떠올린 그림은 직관적이었습니다. 유저 연령대별로 상품 점수를 매기고, 그 점수 내림차순으로 카테고리 리스트를 다시 정렬한다. 하지만 이 그림에는 세 가지가 걸렸습니다.

- **두 점수 체계가 충돌합니다.** 카테고리 리스팅은 이미 검색 엔진의 점수로 순서가 정해집니다. 그 위에 연령대 정렬을 한 겹 얹으면, 인기·시즌·판매량이 녹아 있던 기존 점수를 연령대 정렬이 통째로 덮어씁니다.
- **비로그인까지 같은 경로를 탑니다.** 정렬 단계 자체를 바꾸면 개인화가 필요 없는 트래픽까지 영향을 받습니다.
- **회귀 피해가 큽니다.** 진입 트래픽이 큰 지면이라, 정렬을 통째로 바꿨다가 문제가 나면 카테고리 탐색 전체가 흔들립니다.

그래서 방향을 틀었습니다. 정렬 단계가 아니라, **점수를 만드는 입력 자체**를 연령대에 맞게 바꾸기로 했습니다. 정렬 로직은 그대로 두고, 그 로직이 읽는 점수만 갈아끼우는 쪽입니다.

## 점수는 검색 시점이 아니라 색인 시점에 박혀 있다

검색 엔진을 들여다보니, 설계를 결정한 사실 하나가 보였습니다. **연령대별 점수는 요청 때 계산되지 않습니다. 색인할 때 상품 문서(document) 안에 미리 들어가 있습니다.**

한 상품 문서에는 전 연령 기준 점수 필드 하나와, 연령대 구간별 점수 필드들이 나란히 박혀 있었습니다.

```
[ 상품 문서 ]
  score_global   : 0.85   ← 전 연령 기준
  score_2534     : 0.91   ← 25~34세
  score_3544     : 0.78   ← 35~44세
  ...
```

이 사실이 설계를 단순하게 만들었습니다. 쿼리가 할 일은 점수 계산이 아니라 "이 요청은 어느 점수 필드를 읽을까"를 고르는 것뿐입니다. 데이터 흐름은 두 개의 독립된 파이프라인으로 나뉩니다.

```
[ 색인 파이프라인 (배치) ]
  외부 추천 점수 → 연령대별 집계 → 캐시 → 색인 시 상품 문서에 필드로 병합

[ 쿼리 파이프라인 (요청 시) ]
  유저 연령대 → 읽을 점수 필드명 결정 → 검색 엔진이 그 필드로 스코어링
```

쿼리 시점에는 추가 DB 조회도, 점수 계산도 없습니다. 이미 구워진 필드를 읽기만 합니다.

## 점수 필드를 연령대별로 바꿔 끼우다

핵심은 단순합니다. 로그인 유저의 연령대가 있으면, 스코어링이 참조할 점수 필드명만 그 연령대 필드로 바꿔 넣습니다.

**개인화 적용 전 (전 유저 동일):**

```python
# 스코어링이 항상 전 연령 기준 점수 필드를 참조
score_params["score_field"] = "score_global"
score_params["use_per_age"] = False
```

**개인화 적용 후 (로그인 + 연령대 보유):**

```python
def apply_age_settings(score_params: dict, age_suffix: str | None) -> None:
    if age_suffix:
        # 예: "2534" → 25~34세 구간 점수 필드
        score_params["score_field"] = f"score_{age_suffix}"
        score_params["use_per_age"] = True
    else:
        # 비로그인 / 연령대 미보유 → 전 연령 기준으로 fallback
        score_params["score_field"] = "score_global"
        score_params["use_per_age"] = False
```

- 연령대 → 필드 suffix 매핑(`"25~34" → "2534"`)만 두면, 정렬 로직은 한 줄도 바뀌지 않습니다.
- 비로그인이나 연령대가 없는 유저는 `age_suffix`가 비어 있으므로 자동으로 전 연령 점수로 떨어집니다. **fallback이 곧 기존 동작**이라, 비로그인 트래픽은 처음부터 영향권 밖입니다.
- 카테고리 페이지 전용 엔드포인트에서만 이 분기를 태웠습니다.

이 설계의 이점은 **건드린 표면이 좁다**는 것입니다. 검색 엔진 쿼리 빌더, 정렬 키, 페이지네이션은 그대로 두고, 점수 필드명이라는 입력 파라미터 하나만 갈아끼웠습니다. 단독 작업이라 누가 결정을 검증해 주지 않는 상황에서, 표면을 좁게 가져가는 선택이 특히 든든했습니다.

## script_score 안에서 일어나는 일

점수 필드명을 넘기면 검색 엔진은 그걸로 무엇을 할까요. 여기서 한 가지를 분명히 짚어둘 만합니다. 이 스코어링은 `function_score`가 아니라 `script_score`입니다. 필터는 점수에 영향을 주지 않는 `filter` 절로만 걸고, 점수는 전부 스크립트가 계산합니다.

최종 점수의 모양은 이렇습니다.

```
최종 점수 = (연령대 점수 + 보조 점수 × 혼합 비율) × 공통 배수
```

- **연령대 점수**는 외부 추천 점수 소스 두 곳(편의상 A·B로 부르겠습니다)의 점수를 더한 값입니다. `use_per_age`가 켜지면 두 소스를 모두, 꺼지면 A의 전 연령 점수 하나만 더합니다.
- **보조 점수**는 기존 스코어링이 쓰던 가중합(가격·시즌·리뷰 등)이고, 여기에 작은 혼합 비율을 곱해 더합니다. 연령대 점수를 주연으로, 기존 점수를 조연으로 둔 구조입니다.
- **공통 배수**는 셀러·브랜드·옵션 관련 보정값입니다.

여기서 두 가지 동작이 회고할 만합니다.

- **점수 없는 상품은 배제가 아니라 강등됩니다.** 어떤 상품 문서에 해당 연령대 점수 필드가 아예 없으면, 스크립트는 그 값을 `0`으로 읽습니다. 그러면 연령대 점수가 0이 되고 보조 점수만 남아 자연히 뒤로 밀립니다. 점수가 없다고 카테고리에서 사라지지는 않습니다.
- **연령대 모드에서는 공식 자체가 살짝 달라집니다.** 카테고리 리스팅은 원래 판매량·가격을 로그로 곱해 인기순을 만들었는데, 연령대 모드에서는 그 항을 뺍니다. 판매량·가격 신호가 이미 연령대 점수 안에 녹아 있어 중복이기 때문입니다.

## 연령대 점수는 미리 만들어 둔다 — 5살을 10살로 접기

요청마다 연령대 점수를 실시간으로 계산하면 진입 트래픽을 감당하기 어렵습니다. 그래서 점수는 배치로 미리 만들어 캐시에 올려 두고, 색인할 때 상품 문서에 병합했습니다.

만들 때 손이 간 부분이 있습니다. 원천 점수는 5살 단위(예: 25~29, 30~34)로 집계되어 있었는데, 서비스에서 쓰는 구간은 10살 단위였습니다. 두 구간을 합쳐야 했습니다.

- 단순 평균으로 합치면, 표본이 거의 없는 구간이 점수를 왜곡합니다.
- 그래서 **최근 조회수로 가중 평균**을 냈습니다. 사람이 많이 본 5살 구간의 점수가 더 크게 반영됩니다.
- 다만 두 구간의 조회수 합이 0이면 가중치를 줄 수 없으므로, 이때만 단순 평균으로 떨어뜨렸습니다.

```python
# 25~29 (점수 0.8, 조회수 1000) + 30~34 (점수 0.4, 조회수 50)
# → 25~34 = (0.8*1000 + 0.4*50) / 1050 ≈ 0.78
#
# 단순 평균이면 0.6 — 조회수 50짜리 구간이 과대 반영됨
# 조회수 합이 0이면 그때만 (0.8 + 0.4) / 2 = 0.6 으로 fallback
```

- 65세 이상은 여러 5살 구간을 하나로 합쳐 단일 구간으로 접었습니다.
- 매핑되지 않는 이상 연령대나 점수가 비어 있는 행은 집계 전에 걸러냈습니다.

이 집계는 배치로 주기적으로 돌려 캐시를 갱신했습니다. 캐시에는 짧은 만료를 둬, 배치가 한두 번 실패해도 직전 점수로 버티게 했습니다. 가중 평균을 택한 건 "사람이 실제로 본 비중"을 신뢰한 결정이었습니다. 표본이 적은 구간이 첫 화면을 끌고 가는 일을 막고 싶었습니다.

## 검색·베스트를 왜 뺐나

개인화는 카테고리 기본 리스팅에만 켜고, 검색과 베스트는 뺐습니다. 이게 단순한 범위 축소가 아니라 구조로 보장되는 게 마음에 들었습니다.

연령대 점수를 더하는 코드는 **스크립트 자체에 박혀 있습니다.** 카테고리 리스팅·베스트 스크립트에는 연령대 점수 함수가 들어 있지만, 검색 스크립트에는 그 함수가 아예 import되어 있지 않습니다. 그래서 검색 경로에 실수로 `age_suffix`를 넘겨도, 그 값을 읽는 라인이 없어 그냥 무시됩니다.

의도 측면에서도 맞습니다. 검색은 키워드라는 명확한 의도가 있고, 베스트는 랭킹이라는 의도가 있습니다. 이미 의도가 또렷한 경로를 연령대로 흔들 이유가 없었습니다.

## 트레이드오프 — 캐시는 여러 갈래로 쪼개지고, 커버리지엔 구멍이 있다

잘 풀린 설계에도 비용은 있습니다. 두 가지를 정직하게 적습니다.

**캐시 파편화.** 검색 결과 캐시 키에는 요청 파라미터가 들어가는데, 여기에 연령대 버킷도 포함됩니다. 그래서 같은 카테고리라도 연령대 구간 수만큼 캐시 엔트리가 갈라집니다. 비개인화 대비 결과 캐시 적중률이 그만큼 분산됩니다. 다만 점수 자체는 색인에 구워져 있어 쿼리 시점의 추가 조회는 없으므로, 파편화는 결과 캐시에만 한정됩니다. 그래도 이건 개인화의 명백한 대가입니다.

**커버리지 구멍.** 연령대 값은 소셜 로그인의 생년 동의에서 계산됩니다. 그 값이 없는 유저 — 비로그인, 생년 미동의, 그 경로를 거치지 않은 가입자 — 는 전부 전 연령 점수로 떨어집니다. 개인화가 실제로 닿는 모수가 생각보다 좁다는 뜻입니다. 설계 당시 이 커버리지를 정량으로 잡아두지 못한 게 아쉽습니다.

## 다시 한다면

- **측정 인프라를 처음부터 같이 깐다.** 개인화 적용 구간과 대조군을 처음부터 나눠 측정했어야 했습니다. 지금 구조로는 연령대 개인화가 카테고리 전환에 준 효과를 깔끔하게 분리해 입증하기 어렵습니다. 설계는 좁고 안전하게 잘 풀렸지만, 정량 입증이 늦었습니다.
- **단일 축의 천장을 처음부터 인정한다.** 연령대가 같아도 취향은 갈립니다. 같은 구간 안에서는 결국 비슷한 첫 화면을 봅니다. 구매 이력·관심사 같은 시그널을 함께 묶는 그림을 처음부터 그렸어야 했습니다. 실제로 구매 이력 기반 개인화는 별도 작업으로 이어서 진행했습니다.
- **커버리지를 끌어올릴 입력 경로를 넓힌다.** 연령대를 채울 경로가 소셜 생년 하나뿐이면, 개인화가 닿는 유저가 늘 제한됩니다. 가입·프로필 단계에서 연령대를 보강할 입력을 같이 설계했어야 했습니다.
