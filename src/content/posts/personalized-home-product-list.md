> **TL;DR.** 홈 상품 구좌는 전 유저에게 같은 리스팅을 보여주고 있었습니다. 가장 강한 구매 신호인 최근 구매 브랜드를 한 번도 쓰지 않고 있었죠. 이걸 앵커로 잡아 "최근 산 브랜드의 인기 상품"을 다시 띄우되, 구매 시점이 가까우면 같은 카테고리를 다시, 멀어지면 다른 카테고리로 넓히고, 추천이 노출 칸을 못 채우면 조건을 단계적으로 풀어 채우는 구좌를 붙였습니다. 비로그인·비구매자는 기존 리스팅 그대로 두었습니다.

대형 패션 커머스의 홈 상품 구좌를 구매 이력으로 개인화한 작업의 회고입니다. 백엔드 1인 단독 설계·구현이었고, 잘 풀린 결정과 늦었던 자리를 같이 적습니다.

## 배경 — 가장 강한 구매 신호를 홈에서 버리고 있었다

- **대상** 홈 화면의 상품 구좌. 노출 대비 전환 기여가 큰 자리입니다.
- **선행 상태** 이 자리가 모두에게 똑같이 동작했습니다. 우리 서비스에서 이미 무언가 산 유저에게도, 처음 온 사람과 같은 화면을 보여주고 있었습니다.
- **역할** 백엔드 1인 단독 설계·구현.

재구매를 끌어낼 장치가 전혀 없었습니다. 구매 이력이라는, 가장 직접적인 단서를 홈에서 버리고 있었던 셈입니다.

## 개인화는 좁은 모수에만 켠다

방향은 단순했습니다. 최근 구매 이력이 있는 유저에게, 그 브랜드의 인기 상품을 홈 구좌에 다시 노출합니다. 설계에서 먼저 정한 건 **누구에게 켜느냐**였습니다.

전 유저에게 개인화를 켜는 대신, 조건을 만족하는 좁은 모수에만 켜고 나머지는 기존 리스팅을 그대로 두기로 했습니다. 이유는 두 가지입니다.

- 기존 리스팅 로직을 건드리지 않아 **회귀 위험이 없습니다.**
- "구매 이력이 있는 유저"라는 좁은 모수에서 가설을 먼저 검증할 수 있습니다.

이 구좌는 클라이언트가 별도 쿼리 파라미터로 호출합니다. 즉 홈 화면이 "이 자리는 재구매 추천 구좌다"라고 명시적으로 요청하면, 서버가 그때만 재구매 경로를 탑니다. 기본 리스팅과 같은 API 위에 분기를 하나 얹은 구조입니다.

```python
def resolve_policy(user_id: int | None) -> RepurchasePolicy | None:
    if user_id is None:
        return None  # 비로그인 → 기존 리스팅으로

    purchase_info = get_last_purchase_with_cache(user_id=user_id)
    if purchase_info is None:
        return None  # 구매 이력 없음 → 기존 리스팅으로

    days = days_since(purchase_info.ordered_at)
    return RepurchasePolicy.from_purchase_info(purchase_info, days)
```

- 정책이 `None`이면 그대로 기존 경로로 떨어집니다. **분기의 기본값이 기존 동작**이라는 점이 이 설계의 안전판이었습니다.
- 마지막 구매 정보는 매 요청마다 다시 집계하면 비싸므로 캐시를 한 겹 뒀습니다(뒤에서 다시 다룹니다).

## 30일과 60일, 두 경계가 추천의 의미를 바꾼다

여기서부터가 이 작업에서 가장 손이 많이 간 부분입니다. "최근 구매 브랜드를 보여준다"는 한 문장이, 마지막 구매가 얼마나 지났느냐에 따라 전혀 다른 추천이 됩니다. 마지막 구매 시점을 기준으로 세 구간으로 나눴습니다.

| 구간 | 1차 카테고리 산출 | 추천의 의미 |
|------|------------------|-------------|
| **30일 이내** | 구매 카테고리 ∩ 시즌 카테고리 (교집합) | "방금 산 그 카테고리"를 다시 — 같은 결의 상품 재구매 유도 |
| **30~60일** | 시즌 카테고리 − 구매 카테고리 (차집합) | "산 건 됐고, 같은 브랜드의 **다른** 카테고리" — 크로스셀 |
| **60일 초과** | 정책 없음 | 너무 오래된 구매. 재구매 추천 미적용, 기존 경로로 |

브랜드는 세 구간 모두 마지막 구매 브랜드 하나로 같습니다. 달라지는 건 **카테고리 필터의 방향**입니다.

- 산 지 얼마 안 됐으면, 방금 본 그 카테고리를 다시 보여주는 게 자연스럽습니다(교집합).
- 시간이 좀 지났으면, 같은 걸 또 들이미는 대신 같은 브랜드의 다른 카테고리로 넓힙니다(차집합).
- 산 지 너무 오래됐으면, 그 브랜드를 다시 미는 건 추천이 아니라 잡음이라고 봤습니다. 이 구간은 아예 개인화 범위 밖으로 두고 기존 경로로 보냈습니다.

같은 데이터(마지막 구매 브랜드·카테고리)를 시간 축 하나로 다르게 해석한 셈입니다. 경계값(30일·60일)은 정확히 "미만"·"이상"으로 코드와 테스트에 못 박았습니다.

## "시즌 카테고리"는 달력이 정한다

위 표에 계속 등장하는 "시즌 카테고리"는 마지막 구매와 무관하게, **현재 달력 월**이 정합니다.

- 봄·여름 구간이면 SS, 가을·겨울 구간이면 FW로 묶습니다.
- 환절기에 해당하는 달은 두 시즌을 모두 허용하는 그레이스 구간으로 뒀습니다.
- 시즌 매핑이 없는 카테고리(사계절 상품 등)는 항상 포함되도록 처리했습니다.

그래서 정책에 들어가는 카테고리는 "마지막 구매 카테고리"가 아니라 **"마지막 구매 카테고리 중 지금 시즌에도 파는 것"**입니다. 11월에 반팔을 샀던 기록으로 한겨울에 반팔을 추천하는 일을 막기 위한 장치입니다. 마지막 구매와 현재 시즌을 교차시켜야 "지금 살 만한" 추천이 됩니다.

## 진짜 문제는 노출 칸을 비우지 않는 것

가장 손이 많이 간 건 추천 정확도가 아니라 **노출 칸을 채우는 일**이었습니다.

"최근 산 브랜드 + 그 카테고리"만으로 구좌를 채우려고 하면, 브랜드가 작거나 시즌이 안 맞을 때 상품이 몇 개 안 나옵니다. 홈 구좌가 절반만 차서 빈 자리가 보이면, 개인화를 안 하느니만 못한 화면이 됩니다.

그래서 목표 개수를 채울 때까지 조건을 단계적으로 넓히는 fallback 파이프라인을 뒀습니다.

```
1차: 마지막 구매 브랜드 + 시즌 카테고리   (가장 좁고 정확)
        │ 목표 개수 미달이면
        ▼
2차: 같은 브랜드 전체 인기              (카테고리 조건 제거)
        │ 그래도 미달이면
        ▼
3차: 같은 카테고리 + 전체 브랜드 인기    (브랜드 조건 제거)
        │
        ▼
   목표 개수를 채우면 즉시 중단
```

```python
def fill_until_enough(result, params, policy, client):
    for step in policy.fallback_steps:
        if len(result.products) >= TARGET_COUNT:
            break  # 목표 개수 채우면 중단
        params.brands = step.brands
        params.categories = step.categories
        supplement = client.search_products(params)
        result = merge_dedup(result, supplement)  # 중복 제거 후 뒤에 append
    return result
```

세 가지를 신경 썼습니다.

- **조기 종료.** 각 단계는 목표 개수를 채우면 즉시 멈춥니다. 1차만으로 충분하면 2·3차는 아예 검색하지 않습니다. 대부분의 요청은 1라운드로 끝납니다.
- **순서 보존 dedup.** 병합은 상품 ID 기준으로 중복을 제거하되, 기존 결과를 앞에 그대로 두고 신규만 뒤에 붙입니다. 그래서 좁고 정확한 1차 결과가 항상 상위에 고정되고, 느슨한 fallback 결과가 그 아래로 깔립니다.
- **전체 노출 방지 안전장치.** 3차에서 브랜드도 카테고리도 비면 필터가 모두 사라져 전 카탈로그 인기 상품이 노출됩니다. 이를 막으려고, 줄 수 있는 카테고리가 하나도 없을 때만 브랜드를 유지해 최소한 브랜드 안에 가뒀습니다.

이 단계 설계가 추천 품질과 노출 완성도 사이의 타협점이었습니다. 정확한 추천을 우선 시도하되, 칸이 빌 바엔 조금 느슨한 추천으로라도 채운다는 우선순위를 코드에 박아 둔 것입니다. 정렬은 단계마다 검색 엔진의 인기순 점수를 그대로 썼습니다 — 추천의 "무엇을"은 fallback 단계가 정하고, "어떤 순서로"는 기존 스코어링에 맡긴 구조입니다.

## 두 개의 개인화 경로 — 재구매와 최다 구매

재구매 구좌를 만들면서 알게 된 건, 이 서비스에 개인화 경로가 이미 하나 더 있었다는 점입니다. 둘은 앵커를 산출하는 방식부터 다릅니다.

| 경로 | 앵커 | 데이터 기준 |
|------|------|-------------|
| **재구매 (이번 작업)** | 마지막 유효 구매 **1건**의 브랜드 + 카테고리 | 전 기간, 구매 시점 정렬 |
| **기존 개인화** | 최근 N일 **최다** 구매 브랜드 1개 | 최근 N일, 결제완료 건 집계 |

재구매 경로가 정책이 없다고 판단하면(비로그인·비구매·60일 초과) 기존 개인화로 자연스럽게 넘어갑니다. 재구매가 "최후 1건 + 카테고리 + 시즌 + fallback"의 정교한 파이프라인이라면, 기존 개인화는 "최근 최다 브랜드 하나로 단순 필터"입니다. 같은 "개인화"라는 단어를 쓰지만 정밀도가 다릅니다.

두 경로를 하나로 합치지 않은 건 의도였습니다. 재구매는 "방금 산 그 결을 다시"라는 좁고 강한 가설이고, 최다 구매는 "이 사람이 자주 사는 브랜드"라는 넓은 가설입니다. 가설이 다르면 코드도 갈라두는 게 검증에 유리하다고 봤습니다.

## 무엇을 "구매"로 칠 것인가

앵커가 되는 "마지막 구매"의 정의도 따로 잡아야 했습니다. 결제만 됐다고 다 구매로 치면 안 됩니다.

- 결제완료·출고·배송중·배송완료·구매확정 등 **유효한 진행 상태만** 구매로 인정했습니다.
- 취소·환불·교환 진행/완료 건은 앵커에서 제외했습니다. 환불한 브랜드를 다시 미는 추천은 역효과입니다.
- 마지막 구매 1건을 조회할 때 주문·상품·브랜드를 한 번에 가져오도록 `select_related`로 묶어 N+1을 막았습니다. 유저당 1건이라 카테고리 조회까지 합쳐 쿼리 두 번이면 끝납니다.

취소·환불을 제외하는 이 한 줄이 추천 신뢰도의 바닥을 만듭니다. 테스트도 "전량 취소면 그 이전 유효 구매로", "전부 취소·환불이면 앵커 없음" 같은 경계를 우선 깔았습니다.

## 캐시는 두 겹 — 6시간과 15분

진입 트래픽이 큰 홈 구좌라, 캐시를 두 층으로 뒀습니다.

- **앵커 메타데이터 캐시 (유저별, 6시간).** 마지막 구매 브랜드·카테고리·시점을 유저별로 캐싱합니다. 6시간 동안은 마지막 구매 조회 쿼리를 건너뜁니다. 단 구매나 취소가 발생하면 커밋 직후 비동기로 해당 유저 캐시를 즉시 무효화합니다. 방금 산 사람이 6시간 묵은 앵커로 추천받는 일을 막기 위해서입니다.
- **검색 결과 캐시 (파라미터별, 15분).** 검색 파라미터가 같으면 유저가 달라도 결과를 공유합니다. fallback 단계가 진행될수록 파라미터가 넓어지므로(브랜드만, 카테고리만), 인기 브랜드일수록 캐시 공유율이 올라가 실제 부하가 완화됩니다.

앵커는 유저마다 다르니 유저별로, 검색 결과는 같은 조건이면 공유 가능하니 파라미터별로. 캐시 키의 단위를 둘로 나눈 게 이 구좌의 부하를 버티게 한 핵심입니다.

## 회고 — 1차가 자주 빗나가는 구조적 이유, 그리고 단일 시그널의 천장

출시 후 이 구좌에서 측정 가능한 매출 기여가 새로 생겼습니다. 없던 노출 자리에서 새 전환 경로가 만들어졌다는 점이 핵심입니다(구체 수치는 비공개).

다만 fallback을 그렇게 정교하게 짜야 했던 데에는 코드를 다시 보고서야 또렷해진 구조적 이유가 있었습니다. **앵커가 들고 오는 카테고리 이름과, 검색 인덱스에 저장된 카테고리 이름의 형태가 어긋날 수 있습니다.** 앵커는 마지막 구매 상품의 카테고리명을 통째로(예: `"반팔/반소매"`) 들고 오는데, 인덱스에는 그게 분할된 토큰(`"반팔"`, `"반소매"`)으로 저장돼 있습니다. 정확 매칭을 걸면 1차가 0건이 날 수 있습니다.

결과가 0건으로 이어지진 않습니다. 2차(브랜드 전체)가 곧바로 채우니까요. 하지만 이건 "1차 적중이 생각보다 자주 빗나가고, fallback이 그걸 메우고 있었다"는 뜻입니다. fallback을 잘 깐 덕에 노출은 멀쩡했지만, 카테고리 이름의 정규화를 양쪽에서 맞췄다면 1차 적중률 자체가 올라갔을 겁니다. 이걸 출시 시점에 정량으로 못 잡은 게 아쉽습니다.

정성적으로는 방향이 갈렸습니다.

- 구매가 잦은 유저에게는 재구매 유도가 분명히 작동했습니다.
- 구매 빈도가 낮은 유저는 같은 브랜드가 계속 떠서 금세 지루해했습니다.

좋았던 판단은 좁은 모수에 이원화로 켜서 회귀 위험 없이 빠르게 띄운 것입니다. 분기의 기본값을 기존 동작으로 둔 안전판이 단독 작업에서 특히 든든했습니다.

아쉬운 판단은 "최근 구매 브랜드"라는 단일 시그널에만 기댄 점입니다. 헤비 구매자에게는 통했지만 라이트 유저에게는 같은 화면의 반복이었습니다. 같은 브랜드가 연속으로 뜨지 않도록 분산 규칙을 처음부터 넣었어야 했고, 구매 이력 외에 연령대·관심사 같은 시그널을 함께 묶었어야 했습니다. 실제로 이 구좌는 라이트 유저 경험을 이유로 피보팅을 준비하고 있습니다. 다양성 보정을 나중 과제로 미룬 게 이번 작업에서 가장 늦은 자리였습니다.
