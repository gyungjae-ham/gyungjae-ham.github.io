> 2024-12 tistory 회고를 5~6년차 백엔드 시각으로 다시 손본 글입니다. 그때는 "전역 변수에 잠깐만 담자" 가 합리적으로 보였습니다. 1년차의 시야 안에 Redis 도, SAML 의 `RelayState` 도 없었기 때문이었습니다.

1년차 시절, **동시 접속 5,000명 규모의 대학 LMS 솔루션** 을 SAML 로 Canvas 와 연결하던 자리에서 시연 사고를 냈습니다. 시연 시작 5분 만에 고객사 인원 20명이 동시 로그인 한 자리에서 *서로 다른 사람의 정보로 로그인되는* 장면이 떴습니다. 그때의 판단은 1년차의 시야 안에서 합리적으로 보였습니다. 지금 보면, 그 판단을 *어디서 멈췄어야 했는가* 가 더 중요한 자리입니다.

## 사건 — 시연 5분 만에 타인 로그인이 떴습니다

고객사 인원 20명이 같은 자리에서 시연용 계정으로 동시 로그인하던 자리였습니다. 5분 만에 *다른 학생의 강의실에 들어간* 화면이 떴고, 시연은 즉시 중단되었습니다. 1년차의 머릿속에는 단 하나의 압박만 남아 있었습니다. **납품이니까 어떻게든 해내야 한다.** 자조나 변명이 끼어들 자리가 아니었습니다.

## 그때의 사고 모델 — "전역 변수에 잠깐만" 이 어떻게 합리적으로 보였는가

SAML 로그인 URL 요청 시점에 강의실 정보 같은 부가 컨텍스트를 받아야 했습니다. Canvas 에서 콜백이 들어올 때는 *어떤 유저인지조차 알려주지 않아*, DB 에 적재해 두고 매칭할 키가 없었습니다.

그래서 떠올린 판단이 **"전역 변수에 잠깐 담아두자"** 였습니다. 1년차의 사고 모델 안에서 그건 *Request scope 의 짧은 캐시* 처럼 보였습니다.

문제는 시야의 한계였습니다. **Redis 도 외부 세션 저장소도 시야 안에 없었습니다.** SAML 의 `RelayState` 같이 *표준이 제공하는 빈자리* 도 활용하지 못했습니다. SAML 흐름을 *덜 알았기 때문* 입니다. 시야가 좁으면 "잠깐만 담아두면 된다" 라는 발상이 합리적으로 보이게 됩니다.

## 임시 보완과 한계

시연 중단 직후 떠올린 보완책은 **"사용 직후 전역 변수를 초기화"** 였습니다. 동시 요청이 들어오면 초기화 전에 다른 요청이 끼어들 수 있어 *사고 빈도만 줄였을 뿐 본질을 해결하지 못했습니다.* 1년차의 시야에서는 그래도 *일단 막는다* 는 의미였습니다.

## 최종 해결 — 클라이언트가 컨텍스트를 들고 다니게

결국 **쿠키에 암호화한 컨텍스트를 담아 클라이언트가 들고 다니게 하는 방식** 으로 갔습니다. Canvas 로 넘어가는 시점에 클라이언트에서 다시 꺼내 인증하니, 요청마다 컨텍스트가 독립적이라 *동시 요청에 영향을 받지 않았습니다.* 타인 로그인 이슈는 그 변경 이후 0건이 되었습니다.

| 방식 | 컨텍스트 위치 | 동시 요청 안전성 | 그때의 시야 |
|---|---|---|---|
| 서버 전역 변수 | 서버 메모리 (공유) | ❌ | 합리적으로 보였음 |
| 서버 메모리 + 즉시 초기화 | 서버 메모리 (공유) | ❌ 경합 잔존 | 시연 직후 임시 보완 |
| 서명·암호화 쿠키 | 클라이언트 | ✅ 요청마다 독립 | 최종 선택 |
| 짧은 TTL Redis (요청 식별자 키) | 서버 외부 저장소 | ✅ 단, 키 설계 필요 | 시야 밖 |
| SAML `RelayState` | SAML 표준이 제공하는 자리 | ✅ 표준 호환 | 시야 밖 |

쿠키 · Redis · `RelayState` 가 같은 결입니다. **컨텍스트가 서버 메모리가 아닌 곳에 있다.** 이 결을 보지 못한 자리가 1년차 시야의 한계였습니다.

## 지금의 질문 순서 — 5~6년차 시각의 재구성

지금 같은 자리에 서면 첫 의심부터 다릅니다. *전역 변수에 요청 정보를 담는다* 는 발상이 보이는 순간, **스레드 간 공유 자원** 의 자리가 즉시 의심에 오릅니다. 이건 *매우 심각한 문제* 입니다. 그 의심이 도구를 정하기 *전에* 옵니다.

도구를 정하는 순서는 다음과 같이 잡힙니다.

1. **요청이 stateless 한가** — 클라이언트가 컨텍스트를 들고 다닐 수 있는가
2. **컨텍스트의 크기·민감도** — 쿠키에 담아도 되는 길이·민감도인가
3. **저장소 후보** — 서명·암호화 쿠키 → JWT → 짧은 TTL Redis → DB
4. **서버 메모리는 *마지막의 마지막* 후보** — 단일 인스턴스가 보장되는 *읽기 캐시* 정도의 자리에만

지금이라면 **Redis** 로 풀었을 것입니다. 짧은 TTL 의 키 하나로 요청 식별자와 컨텍스트를 묶고, Canvas 가 다시 요청해 올 때 그 키로 꺼내 쓰는 방식입니다. 또는 SAML 흐름이라면 `RelayState` 가 *표준이 정확히 그 자리를 위해 만든 칸* 이니, 이쪽이 더 깔끔합니다.

그리고 더 중요한 자리. *전역 변수에 잠깐만 담자* 라는 발상이 떠오르는 순간, **그 자체에서 멈췄어야 했습니다.** 도구를 찾기 *전에*, 서버 메모리에 요청 컨텍스트를 두는 발상 자체가 *이미 잘못된 자리* 입니다.

## 같은 결의 다른 자리

요청과 응답 사이의 컨텍스트가 *어디에 살아야 하는가* 라는 질문은, 이 사고 뒤에도 여러 자리에서 같은 결로 만났습니다.

- **OAuth `state`** — 표준 패턴이 정확히 같은 결입니다. 사용자가 들고 다니거나 외부 저장소가 들고 있어야 합니다. 제공자가 아닌 *사용자* 입장에서 그 결을 익혔습니다.
- **주문·결제 진행 상태** — 진행 중인 주문은 DB 의 `status` 컬럼이 단일 진실입니다. 서버 메모리는 *진행 컨텍스트* 의 집이 아닙니다.
- **비동기 작업 추적** — Celery 워커의 task 명이 작업 컨텍스트의 키입니다. 작업 식별자가 컨텍스트의 위치를 정합니다.

세 자리의 결은 같습니다. *서버 메모리는 요청·작업 단위 컨텍스트의 집이 아닙니다.* 클라이언트가 들고 다니거나, 외부 저장소가 들고 있거나, 영속 저장소의 단일 컬럼이 들고 있어야 합니다.

## 정리

전역 변수에 요청 정보를 잠깐 담자는 발상이 떠오르는 순간이 신호입니다. *저장소가 마땅치 않다* 가 아니라 *서버 메모리로 요청 컨텍스트를 들고 가려고 한다* 는 발상 자체가 멈춤의 자리입니다. 1년차에는 시야가 좁아 그 신호를 보지 못했습니다. 5~6년차의 자리에서 다시 보면, 도구를 정하기 전에 *전역 변수 자체에서 멈췄어야 한다* 가 진짜 결론입니다. 서버 메모리는 요청 단위 컨텍스트의 집이 아닙니다.
