> 2024-12 tistory 회고를 5~6년차 백엔드 시각으로 다시 손본 글입니다. 그때는 "전역 변수에 잠깐만 담자" 가 합리적으로 보였습니다. 1년차의 시야 안에 Redis 도, SAML 의 `RelayState` 도 없었기 때문이었습니다.

1년차 시절 운영하던 대학 LMS 솔루션의 시연 자리에서 사고를 냈습니다. 동시 접속 5,000명 규모의 솔루션을 SAML 로 Canvas 와 연결하는 자리였고, 시연이 시작된 지 5분 만에 서로 다른 사람의 정보로 로그인되는 장면이 떴습니다. 고객사 인원 20명이 같이 앉아 있던 시연자리였고, 즉시 중단되었습니다.

그때 내린 판단은 1년차의 시야 안에서는 합리적으로 보였습니다. 지금 다시 들여다보면 그 판단을 어디서 멈췄어야 하는가 가 더 중요한 자리고, 이 글은 거기서 출발합니다.

## 그때의 사고 모델 — "잠깐만 담아두자"

SAML 로그인 URL 을 요청하는 시점에 강의실 정보 같은 부가 컨텍스트를 받아야 했습니다. 다만 Canvas 가 다시 서버로 콜백할 때는 어떤 유저가 요청한 것인지조차 알려주지 않아, DB 에 적재해 두고 매칭할 키가 마땅치 않았습니다. 그 자리에서 떠올린 발상이 "전역 변수에 잠깐 담아두자" 였습니다.

지금 보면 우스운 선택이지만, 그때 머릿속을 짚어보면 결이 보입니다. Redis 같은 외부 세션 저장소는 시야 안에 들어와 있지 않았고, SAML 인증을 제대로 활용할 만큼 표준을 들여다보지 못했습니다. 그래서 잠시 어딘가에 담아두려고 했고, 손에 잡힌 자리가 서버 메모리였습니다. 시야가 좁으면 "잠깐만 담아두면 된다" 라는 발상이 합리적으로 보이게 됩니다.

당연하게도 개발 단계에서 그 로직은 너무 잘 동작했습니다. 테스트는 본인 혼자 접속해서 돌리는 자리였으니 전역 변수의 공포를 느낄 수 없었습니다. 그게 시연 자리에서 처음 터졌습니다. 5분 만에 다른 사람의 강의실 정보로 로그인되는 화면이 떴고, 고객사 인원 20명이 그 장면을 함께 봤습니다. 1년차 머릿속에는 단 하나의 압박만 남아 있었습니다. 납품이니까 어떻게든 해내야 한다.

그 자리에서 머리를 굴려 떠올린 1차 보완은 "사용 직후 전역 변수를 즉시 초기화" 였습니다. 사고 빈도는 줄었지만, 초기화 직전에 다른 요청이 끼어드는 자리가 그대로 남았습니다. 본질을 해결하지 못한 우회였고, 1년차 시야 안에서는 그래도 일단 막는다 는 의미였습니다. 결국 쿠키에 암호화한 컨텍스트를 담아 클라이언트가 들고 다니게 하는 방식으로 갔습니다. Canvas 로 넘어가는 시점에 클라이언트가 컨텍스트를 다시 꺼내 인증하니 요청마다 컨텍스트가 독립적이라 동시 요청에 영향을 받지 않았고, 타인 로그인 이슈는 그 변경 이후 0건이 되었습니다.

## 같은 자리의 다른 답들 — 결은 같습니다

지금 같은 자리에 서면 후보를 더 펼쳐 비교합니다. 핵심은 컨텍스트가 서버 메모리가 아닌 곳에 있어야 한다는 한 줄입니다.

| 방식 | 컨텍스트 위치 | 동시 요청 안전성 | 그때의 시야 |
|---|---|---|---|
| 서버 전역 변수 | 서버 메모리 (공유) | ❌ | 합리적으로 보였음 |
| 서버 메모리 + 즉시 초기화 | 서버 메모리 (공유) | ❌ 경합 잔존 | 시연 직후 임시 보완 |
| 서명·암호화 쿠키 | 클라이언트 | ✅ 요청마다 독립 | 최종 선택 |
| 짧은 TTL Redis (요청 식별자 키) | 서버 외부 저장소 | ✅ 단, 키 설계 필요 | 시야 밖 |
| SAML `RelayState` | SAML 표준이 제공하는 자리 | ✅ 표준 호환 | 시야 밖 |

쿠키 · Redis · `RelayState` 가 같은 결입니다. 컨텍스트가 클라이언트나 외부 저장소에 살고, 서버 인스턴스끼리 공유되는 메모리에 묶이지 않습니다. 이 결을 보지 못한 자리가 1년차 시야의 한계였습니다.

## 지금의 질문 순서

지금 같은 자리에 서면 첫 의심부터 다릅니다. 전역 변수에 요청 정보를 담는다 는 발상이 보이는 순간 스레드 간 공유 자원의 자리가 먼저 떠오르고, 그게 매우 심각한 문제 라는 감각이 같이 옵니다. 그 의심이 도구를 정하기 전에 옵니다.

도구를 정하는 순서는 그 다음입니다. 요청이 stateless 한가, 그래서 클라이언트가 컨텍스트를 들고 다닐 수 있는가 를 먼저 묻습니다. 다음으로 컨텍스트의 크기와 민감도를 봅니다. 짧고 민감하지 않다면 서명·암호화 쿠키 또는 JWT, 그보다 길거나 서버에서만 봐야 한다면 짧은 TTL Redis, 영속이 필요하면 DB 의 status 컬럼 으로 내려갑니다. 서버 메모리는 마지막의 마지막 후보고, 단일 인스턴스가 보장되는 읽기 캐시 정도에만 둡니다.

지금이라면 Redis 로 풀었을 가능성이 높습니다. 짧은 TTL 의 키 하나로 요청 식별자와 컨텍스트를 묶고 Canvas 콜백 시점에 그 키로 꺼내 쓰는 방식이 가장 직선적입니다. SAML 흐름이라면 `RelayState` 가 정확히 그 자리를 위해 표준이 만들어둔 칸이니 그쪽이 더 깔끔합니다. 다만 더 중요한 자리는 따로 있습니다. 전역 변수에 잠깐만 담자 라는 발상이 떠오르는 순간, 도구를 찾기 전에 그 자체에서 멈췄어야 합니다.

## 같은 결의 다른 자리

요청과 응답 사이의 컨텍스트가 어디에 살아야 하는가 라는 질문은 이 사고 뒤에도 여러 자리에서 같은 결로 다시 만났습니다. OAuth `state` 가 같은 결입니다. 표준이 정확히 그 결을 위해 만든 칸이고, 제공자가 아닌 사용자 입장에서 그 결을 익혔습니다. 결제와 주문의 진행 상태도 결이 같습니다. 진행 중인 주문은 DB 의 `status` 컬럼이 단일 진실이고, 서버 메모리는 진행 컨텍스트의 집이 아닙니다. 비동기 작업 추적도 같습니다. Celery 워커의 task 명이 작업 컨텍스트의 키가 되고, 작업 식별자가 컨텍스트의 위치를 정합니다.

세 자리 모두 결이 같습니다. 서버 메모리는 요청·작업 단위 컨텍스트의 집이 아닙니다. 클라이언트가 들고 다니거나, 외부 저장소가 들고 있거나, 영속 저장소의 단일 컬럼이 들고 있어야 합니다.

## 정리

전역 변수에 요청 정보를 잠깐 담자는 발상이 떠오르는 순간이 신호 입니다. 저장소가 마땅치 않다 가 아니라 서버 메모리로 요청 컨텍스트를 들고 가려고 한다 는 발상 자체가 멈춤의 자리고, 1년차 시야로는 그 신호를 보지 못했습니다. 5~6년차 자리에서 다시 보면 도구를 정하기 전에 전역 변수 자체에서 멈췄어야 한다 가 진짜 결론이고, 서버 메모리는 요청 단위 컨텍스트의 집이 아니라는 한 줄이 그 뒤에 남습니다.
