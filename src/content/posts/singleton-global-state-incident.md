> 2024-12 에 tistory 에 정리한 회고를 5~6년차 백엔드 시각으로 다시 손본 글입니다. 그때는 "전역 변수를 잠깐만 쓰면 괜찮다" 고 봤지만, 운영을 거치고 보면 그 판단은 **"요청 컨텍스트를 잘못 든 자리"** 로 다시 읽힙니다.

대학 LMS 솔루션을 SAML 인증으로 Canvas 와 연결하던 시절의 이야기입니다. 사고는 시연 자리에서 터졌고, 그 사고가 가르친 건 결국 **"서버 측 상태로 요청 컨텍스트를 들고 가지 않는다"** 한 줄이었습니다.

## 사고 — 시연 자리에서 다른 사용자로 로그인되었습니다

유저가 SAML 로그인 URL 을 요청한 시점에 강의실 정보 같은 부가 컨텍스트를 받아야 했습니다. Canvas 에서 다시 우리 서버로 요청이 들어올 때는 어떤 유저인지조차 알려주지 않았기 때문에, DB 에 적재해 두고 매칭할 수도 없었습니다.

그 자리에서 내린 판단은 **"전역 변수에 잠깐 담아두고 다음 콜백에서 꺼내자"** 였습니다. 동시에 한 명만 테스트하던 개발 환경에서는 너무 잘 동작했습니다. 동시 접속이 없으면 전역 변수의 공포는 보이지 않습니다.

문제는 시연이었습니다. 시연자 여러 분이 동시에 로그인하니, 마지막에 전역 변수를 덮어쓴 사람의 정보로 모두가 로그인되었습니다. 떠올리고 싶지 않은 장면이었습니다.

## 해결 시도 — "초기화" 로는 못 막습니다

처음 떠올린 보완책은 **"사용 직후 전역 변수를 초기화"** 였습니다. 동시 요청이 있으면 초기화 전에 다른 요청이 끼어들 수 있어, 사고 빈도만 줄였을 뿐 본질을 해결하지 못했습니다.

최종 해결은 **쿠키에 암호화한 컨텍스트를 담고, Canvas 로 넘어가는 시점에 클라이언트에서 다시 꺼내 인증하는 방식** 이었습니다. 이 변경 후에는 타인 로그인 이슈가 0건이 되었습니다.

| 방식 | 컨텍스트 위치 | 동시 요청 안전성 |
|---|---|---|
| 서버 전역 변수 | 서버 메모리(공유) | ❌ |
| 서버 메모리 + 즉시 초기화 | 서버 메모리(공유) | ❌ 경합 잔존 |
| 서명/암호화 쿠키 | 클라이언트 | ✅ 요청마다 독립 |
| 짧은 TTL Redis (요청 식별자 키) | 서버 외부 저장소 | ✅ 단, 키 설계 필요 |

## 그래서 어떻게 시작하는가

지금 다시 보면, 사고의 진짜 신호는 "저장할 곳이 마땅치 않다" 가 아니라 **"요청과 응답 사이의 컨텍스트를 서버 메모리로 들고 가려고 한다"** 였습니다. 컨텍스트는 클라이언트가 들고 다니거나(쿠키·서명된 토큰), 짧은 TTL 의 외부 저장소를 거치는 게 기본입니다.

전역 변수에 요청 정보를 넣고 싶어진다면, **그 자체로 요청-응답 모델이 잘못 잡혔다는 신호 입니다.** 저장소가 아니라 *전달 방식* 을 먼저 의심해야 합니다.
