# 보이스 가이드

블로그 글 작성 시 이 가이드를 따른다. AstroPaper 블로그(`gyungjae-ham.github.io`)의 모든 신규 글에 적용한다.

분석 대상은 velog `@hiyeeluca`의 11편(2023년 6월 ~ 2024년경 작성)이며, Spring Boot / Spring Security / JPA / 테스트 / 아키텍처 학습 노트 위주이다.

---

## 한 줄 정체성

> **5~6년차 백엔드 개발자가 학습한 개념을 정중한 설명조로 정리하면서, 출처가 분명한 문구는 따옴표로 인용하고 코드로 곧장 증명하는, 군더더기 없는 학습 노트형 톤.**

즉 "튜토리얼"보다는 "스터디 노트"에 가깝다. 친근한 농담이나 의인화는 거의 없고, 단정한 설명과 정의 + 코드 + 흐름도(글로 풀어쓴 단계) 조합이 핵심이다.

---

## 어조와 인칭

### 어미 — `~합니다 / ~입니다` 기조의 정중 설명체
- 11편 전체에서 종결어미는 `~합니다 / ~입니다 / ~됩니다` 가 **압도적 다수**(95% 이상).
- `~한다 / ~이다` 같은 문어체는 거의 등장하지 않음.
- 명사형 종결(`~함 / ~임`)은 코드 주석이나 짧은 bullet 보조 설명에서만 드물게.
- 예시:
  - "클라이언트에서 예외처리를 동일하게 관리할 수 있도록 통일된 `ErrorResponse`를 반환**합니다**"
  - "추천 설정값은 100~1000 사이로, DB와 애플리케이션 부하를 고려하여 결정**합니다**"
  - "`ThreadLocal`에 저장되어 아무 곳에서나 참조가 가능하도록 설계되어 있**습니다**"

### 인칭 — 거의 무인칭 / 1인칭 등장 시 "저"
- "저는 / 제가 / 내가" 같은 1인칭 표현은 11편 중 **거의 등장하지 않음**.
- 등장하는 경우에도 "저자는" 정도로 객관화되며, "나"는 쓰지 않음.
- 독자 호명("여러분", "여러분이") 도 **없음**. 강의체가 아니라 노트체이기 때문.
- 권유형은 "~해봅시다 / ~해보자" 가 아주 가끔. 예: "예를 들어 상품을 여러 개 주문한 상황을 생각해**봅시다**".

### 룰
- 기본은 정중 평서체(`~합니다`).
- 1인칭이 필요한 경우에만 "저"를 쓴다. "내가 / 우리가"는 쓰지 않는다.
- 독자에게 직접 말 거는 호명은 피한다. 대신 "이때 / 이런 경우" 같은 상황 지시어로 대체한다.

## 도입부 패턴

11편을 보면 도입부는 **세 가지 패턴** 중 하나로 시작한다. 새 글을 쓸 때 이 셋 중 하나를 고르면 톤이 맞는다.

### 1. **시리즈/맥락 안내문 (인용 블록)** — 가장 자주
인용 블록(`>`)으로 짧게 "이 글이 어느 시리즈의 어디쯤이며, 어떤 전제 위에서 쓰였는지" 알린다.
- 예: "> 이 시리즈는 TDD를 숙달하기 전에 TEST 자체에 대한 이해를 높이기 위한 학습 시리즈입니다"
- 예: "> 이 시리즈는 Spring Security에 대한 학습을 한 후 기록하는 시리즈입니다. 해당 시리즈는 Spring Security 5.7 버전이므로 현재 6.x 버전과 설정방법은 다르지만 개념적 이해는 같으므로 5.7로 진행했습니다."
- 예: "> cors에 관한 설정 내용 정리글입니다. 추후 내용이 추가될 수 있습니다"

특징: **버전·전제·갱신 가능성**을 솔직하게 명시. 학습 노트의 한계를 숨기지 않음.

### 2. **개념의 곧장 정의** — 두 번째로 자주
도입 인사나 동기 부여 없이 첫 H2 / bullet 첫 줄에서 바로 정의를 던진다.
- 예: "- `결합`과 같은 개념입니다, 다른 객체의 함수를 사용하는 상태를 뜻합니다"
- 예: "- 클라이언트에서 예외처리를 동일하게 관리할 수 있도록 통일된 `ErrorResponse`를 반환합니다"

특징: "왜 알아야 하는지" 같은 동기 부여 문장이 거의 없다. 결론·정의를 먼저 던지고 코드로 곧장 들어간다.

### 3. **기존 코드의 문제 진단** — 리팩토링 글 한정
"개선 전 / 문제점 / 개선 후" 흐름으로 시작.
- 예: "기존 `UserService`의 `create` 메서드는 사용자 생성과 함께 이메일 인증 코드 전송 로직을 직접 처리하고 있었습니다. 문제점은 서비스 레이어가 `JavaMailSender`에 직접 의존하고 있다는 것입니다."

특징: 코드부터 보여주고, 그 다음 진단 → 개선.

## 본문 구조

### 소제목 깊이
- **H2(`##`) 가 주력**. 한 글에 5~10개의 H2가 흐름의 뼈대.
- H3(`###`)는 한 H2 안에서 세부 항목을 나눌 때만 사용. 깊이는 거의 H3까지만, H4 이하는 사용하지 않음.
- 예: 스프링 시큐리티 글 — `## FilterChainProxy` → `## Authentication` → `## SecurityContextHolder, SecurityContext` → `### SecurityContext` / `### SecurityContextHolder`

### bullet vs 단락
- **bullet이 단락보다 압도적으로 많다.** 11편 중 9편이 bullet 비중 60% 이상.
- 정의·항목·단계는 모두 bullet. 단락 형태의 산문은 도입 인용문, 진단 문장, 결론 한 줄 정도에 한정.
- bullet 안에서 `**굵게**`로 핵심 키워드를 강조하는 패턴이 일관됨.
  - 예: "- **기본값:** true"
  - 예: "- **인증 시**", "- **인증 후**"

### 코드 블록
- 거의 모든 글에 `java` 코드 블록이 1개 이상.
- 코드 블록은 **before/after 비교**로 자주 등장. 헤더에 `**적용 전:**` / `**적용 후:**`, 또는 `## 개선 전` / `## 개선 후` 같은 명시적 라벨을 붙임.
- 코드 안 주석으로 설명을 깔기보다, 코드 블록 **위·아래의 bullet으로 설명**하는 편을 선호.

### 인용(`>`) 활용
- 정의를 따옴표로 묶거나 인용 블록으로 처리하는 경우가 잦다.
- 예: `**SOP(Same Origin Policy)**: "어떤 출처에서 다른 출처의 리소스를 사용하는 것을 제한하는 보안 방식"입니다`
- 의미: "이 정의는 내 말이 아니라 정설이다"라는 출처감을 살리는 장치.

### 표
- 11편 중 표(`|`)를 적극적으로 쓴 글은 거의 없음. 정의·비교는 bullet으로 처리.
- 새 블로그에서 표를 도입한다면 **새로운 도구**가 될 수 있음(`## 센스 보강` 참고).

## 자주 쓰는 표현·어휘 사전

| 표현 | 의미·맥락 | 빈도 |
|---|---|---|
| `~합니다 / ~됩니다 / ~입니다` | 기본 어미. 사실 진술. | 매우 높음 |
| `~을 뜻합니다 / ~을 의미합니다` | 용어 정의 직후. | 높음 |
| `~을 담당하는 / ~을 처리하는` | 컴포넌트 역할 설명. | 높음 |
| `~을 위임합니다` | 책임 전달. Spring Security 글에서 특히. | 높음 |
| `~할 때 사용됩니다` | API·설정·어노테이션 도입. | 높음 |
| `~이고, ~입니다` | 두 사실을 한 문장에 연결. | 높음 |
| `~을 두어 / ~을 분리하여` | 아키텍처 변경 의도 설명. | 중간 |
| `~의 신호입니다 / ~알려주는 것입니다` | 테스트·설계 진단 결론. | 중간 (테스트 시리즈) |
| `~가 우선되어야 한다는 뜻입니다` | 원칙 정리. | 중간 |
| `이 부분이 없다면 ~ 실패하게 됩니다` | 실패 케이스 경고. | 중간 |
| `예를 들어 ~을 생각해봅시다` | 비유·예시 진입. (드물게 등장하는 권유형) | 낮음 |
| `~한 부분이 있습니다 / ~한 경우가 있습니다` | 일반화·관찰. | 낮음 |

### 어휘 습관
- **인용 따옴표**: 정의나 핵심 문구를 큰따옴표(`"..."`)로 감싸는 습관이 강함. 예: `"가장 먼저 생각해야 하는 건 Use case이고, 도메인들과 도메인들의 관계를 우선 고려"`
- **백틱 인라인 코드**: 클래스·메서드·어노테이션·필드명은 반드시 백틱(`` ` ``)으로 감쌈. 예: `` `@RestControllerAdvice` ``, `` `BussinessException` ``, `` `SecurityContextHolder.getContext().getAuthentication()` ``.
- **영문 용어 보존**: Authentication, Filter, Repository, Entity, Use case 등은 **한글 번역 없이 영문 그대로**. 단, "인증/인가/의존성/역전/주입" 같은 용어는 한글이 1순위.

## 코드 인용 스타일

### 라벨 패턴
- **`**적용 전:**` / `**적용 후:**`** — 한 글 안에서 짧게 비교할 때.
- **`## 개선 전 [컴포넌트명]` / `## 개선 후 [컴포넌트명]`** — H2 레벨 비교가 필요한 큰 리팩토링에서.
- **`문제 코드:` / `개선된 코드:`** — 짧은 학습 예시에서. ("의존성과 Testability" 글 패턴)

### 코드 위·아래 설명 방식
- 코드 블록 **위**: 어떤 상황인지 한 줄로 짧게 정리.
- 코드 블록 **아래**: bullet으로 핵심 줄을 짚어줌. "~는 ~에 의존하고 있는 상태입니다" 식으로 코드의 한 부분을 해설.
- 코드 안 주석은 최소화. 주석은 `// 1. ...`, `// 2. ...` 같은 단계 번호용으로만.

### 강조 패턴
- bullet 첫머리에 `**굵게**`로 키워드. 예: `- **principal**: 사용자 아이디 혹은 User 객체를 저장`
- 코드 안에서는 특별한 하이라이트(`>`, 색상)를 쓰지 않음. 굵게는 본문에서만.

## 마무리 패턴

마무리는 11편 모두에서 **매우 짧다.** "이상으로 글을 마치며..." 같은 닫는 인사가 없다. 패턴은 셋:

### 1. **마지막 H2가 자연스럽게 결론** (가장 자주)
- 예: 에러 처리 글 → 마지막이 `## spring-boot-starter-validation` 으로 그냥 끝.
- 결론을 따로 두지 않고 마지막 항목 설명이 끝나면 글이 끝난다.

### 2. **`## 결과` 또는 `## 결론` 한 단락** (리팩토링·실험 글)
- 예 ("Service 소형 테스트로 전환"): "테스트가 `UserService`의 로직에만 집중하게 되어 진정한 '소형 테스트'가 완성되었으며, 실행 속도는 중형 테스트와 완전히 다른 수준으로 향상됩니다."
- 예 ("외부 연동을 다루는 방법"): "이 접근 방식은 인터페이스를 통해 의존성을 역전시켜, 실제 외부 연동 없이 테스트 가능한 구조를 만들어냅니다."
- 특징: 한 문장 또는 두 문장. "왜 이 변경이 가치 있었는가"를 짧게.

### 3. **`## 개선 전 추가될 사항` 등 번호 매긴 TODO 리스트** (탐색 중인 주제)
- 예 ("레이어드 아키텍처의 문제점과 해결책") — 번호 1~8로 다음에 더 다뤄볼 주제를 나열.
- 특징: 학습 진행을 솔직히 드러냄. "여기까진 정리했고, 다음은 이걸 더 봐야 한다"는 메모.

### 시리즈 링크
- velog 자체가 시리즈 네비게이션을 제공하므로, 글 본문에 명시적 "다음 글 보기" 링크가 들어간 적은 없음. (AstroPaper에서는 직접 추가 권장 — `## 센스 보강` 참고)

## 길이 감각

| 글 유형 | 대략적 길이 | 예시 |
|---|---|---|
| 짧은 정리형 | 300~500 단어 / H2 4~6개 | Error 처리, CORS, JPA 성능 최적화 |
| 중간 학습형 | 600~1000 단어 / H2 6~10개 | 의존성과 Testability, 개념·대역 |
| 긴 아키텍처형 | 1500~2500 단어 / H2 8~15개 + H3 | 스프링 시큐리티 주요 아키텍처, 레이어드 아키텍처 문제점 |

### 단락·문장 호흡
- **bullet 한 줄은 평균 한 문장**. 두 절 이상 길어지지 않음.
- 산문 단락이 등장할 때도 **2~4문장**이 최대치. 그 이상은 잘 보이지 않음.
- 빈 줄(`\n\n`)로 H2 사이를 분리하지만, 코드 블록 앞뒤로도 빈 줄을 두어 호흡을 확보.

## 사용자만의 습관

### 어미 일관성
- 한 글 안에서 어미가 거의 흔들리지 않음. `~합니다` 로 시작하면 끝까지 `~합니다`. `~한다`와 섞이는 일이 거의 없음.

### 영문 용어 표기
- 클래스명·메서드명·어노테이션·DI 컨테이너 키워드는 **백틱 인라인 코드**로.
- 책에서 가져온 개념(Use case, Test Double, BDD, SUT)은 **영문 원형 그대로**, 처음 등장 시 한글 풀이 병기. 예: `BDD (Behaviour Driven Development)`, `SUT (System Under Test)`
- DI, DIP, DRY, DAMP, OSIV, SOP, CORS, CSRF 같은 약어는 그대로 사용.

### 오타·표기 습관 (관찰된 그대로)
- 일부 글에서 `Bussiness` 같은 오타가 그대로 등장 (의도된 코드 이름이 아니라 단순 오타로 추정). → **새 블로그에서는 `Business`로 교정 권장**.
- 단어 사이 공백 처리 자체는 일관됨 (영문/한글 사이 공백 거의 항상 둠).

### 출처감 표현
- 강의·책·시리즈에서 가져온 문구는 큰따옴표(`"..."`)로 감싼다. 자기 문장으로 환원하지 않고 그대로 살림.
- 인용 블록(`>`)도 같은 용도로 쓰임.

### "신호" 메타포
테스트 시리즈 글에서 반복 등장하는 표현이 있다:
- "테스트가 우리에게 보내는 신호입니다"
- "테스트가 설계에 문제가 있다고 신호를 보내고 있는 부분"
- "이것은 테스트가 알려주는 것이니"

→ 코드/테스트를 **의인화하지 않으면서도 "신호"로 비유**하는 습관. 새 블로그에서도 이 결을 살리면 일관됨.

## 피하는 표현·패턴

이 사용자의 글에는 **거의 등장하지 않는** 표현들. 새 글에서도 회피.

- **이모지 사용 없음** (제목·본문 모두). 정중·기술적 톤 유지 목적.
- **느낌표(!) 거의 없음**. 강조는 `**굵게**`로 처리.
- **농담·자조 거의 없음**. "(예전에 이걸 몰라서 새벽 3시에 디버깅했다)" 같은 사적 메모는 11편 통틀어 0건.
- **독자 호명 없음**: "여러분", "여러분도" 같은 표현 사용 안 함.
- **장황한 도입**: "오늘은 ~에 대해 알아보겠습니다" 같은 인사형 도입 사용 안 함.
- **밈·짤·이모티콘**: 본문에 없음.
- **느낌형 형용사 남용 없음**: "정말", "엄청", "굉장히" 같은 강조 부사는 거의 없음. 정보를 그대로 전달.
- **물음표 도입**: "왜 그럴까요?" 식 질문형 도입 거의 없음.

## "센스"를 더할 지점

사용자가 "글을 좀 더 센스있게 썼으면 좋겠다"고 명시했다. 11편 기존 톤을 깨지 않으면서 더해볼 만한 요소들이다.

### 1. 도입부에 **한 줄짜리 맥락 훅** 추가
기존 도입은 "시리즈 안내 인용문" 또는 "곧장 정의"로 매우 건조하다. 첫 줄에 **현실 문제 한 줄**을 추가하면 같은 정중체로도 흡인력이 생긴다.
- 예 (Before): "## ErrorCode 에러 처리 시 반환할 Http Status 값, 에러코드, 에러메시지를 관리하는 Enum 클래스로 구성됩니다."
- 예 (After): "프론트와 서버가 에러 응답 포맷을 두고 매번 핑퐁을 친 적이 있다. 통일된 `ErrorResponse`를 두면 그 대화가 사라진다."
- 톤 규칙: 정중체 유지. "~친 적이 있다 / ~사라진다" 같은 평서체로 한 줄.

### 2. 마무리에 **한 줄 메타 코멘트** 추가
기존 마무리는 짧거나 없다. AstroPaper 새 블로그에서는 마지막에 "이 다음에 더 파볼 것" 한 줄을 두면 시리즈 연결성이 살아난다.
- 예: "다음 글에서는 이 구조를 production 환경에 옮겨, MDC 기반 트레이싱과 함께 묶어본다."
- 예: "여기까지가 5.7 기준이다. 6.x로 옮기면서 같은 흐름이 어디서 바뀌는지는 별도 글로 정리한다."

### 3. **표·다이어그램**의 선택적 도입
11편에는 표가 거의 없고 다이어그램도 본문에 들어간 적이 적다. 다음 두 경우에만 도입:
- **비교**(어노테이션 vs 어노테이션, before vs after 옵션이 3개 이상): 표가 bullet보다 명료.
- **흐름**(Filter chain, 인증 흐름, 트랜잭션 propagation): mermaid 시퀀스 다이어그램 1개로 본문 길이를 줄임.
- 단, 표·다이어그램은 **bullet을 보조**하는 역할. 본문 주력은 여전히 bullet.

### 4. **"신호" 메타포 의식적으로 살리기**
테스트 시리즈에서 등장한 "테스트가 보내는 신호" 표현은 이 사용자만의 식별 표지다. 새 블로그 글에서도 비슷한 결로 한 줄씩 끼워 넣으면 톤이 일관된다.
- 예: "이 N+1 쿼리는 ORM이 보내는 신호다 — 연관 로딩 전략을 다시 들여다봐야 한다."

### 5. **번호 prefix 시리즈 강화**
velog에서 이미 `1. spring-security`, `2. 개념, 대역`, `3. 의존성과 Testability` 식으로 번호를 붙였다. AstroPaper에서는 **slug + frontmatter `series` 필드**로 시리즈를 명시하고, 본문 첫 인용 블록에 "이 글은 X 시리즈 N편"이라고 짧게 안내하면 기존 습관과 자연스럽게 맞물린다.

### 피해야 할 "센스"
- 과한 농담·자조: 기존 톤과 충돌.
- 이모지: 한 글당 0~1개 이내. 굳이 넣을 필요 없음.
- 의인화 ("Spring이 화났다", "JPA가 슬퍼한다"): 사용자 결과 멀다.

## 예시: 사용자 톤을 살린 단락 변환

분석한 톤을 살려 다음 한 단락을 작성한다.

### 원본 (밋밋함)
> "JPA에서 N+1 문제는 fetch join으로 해결할 수 있다. 또는 EntityGraph를 사용해도 된다."

### 사용자 톤 변환
> 한 명의 회원이 주문한 `Order`를 조회한 뒤, 각 `Order`의 `orderItems`를 출력하려고 했을 때 쿼리가 N+1번 나갔던 경험이 있습니다. 이때 떠올릴 수 있는 해결책은 두 가지입니다.
>
> - **`fetch join`**: JPQL에 `join fetch`를 명시하여 연관 엔티티를 한 번의 쿼리로 함께 조회합니다. 페이징과 함께 쓸 때는 메모리에서 페이징을 수행한다는 경고가 뜨므로 주의가 필요합니다.
> - **`@EntityGraph`**: Repository 메서드에 `@EntityGraph(attributePaths = "orderItems")`를 선언해 fetch 전략을 외부에서 지정합니다. JPQL을 새로 작성하지 않아도 되어 좋습니다.
>
> 두 방법 모두 "지연 로딩 + 반복 조회"라는 N+1의 본질을 깨는 같은 약입니다. 다만 **fetch join은 쿼리에 박혀 있고, EntityGraph는 메서드 시그니처에 박혀 있다는** 점이 다릅니다. 페이징·정렬 조건이 자주 바뀌는 화면이라면 `@EntityGraph`가 손에 편했고, 단발성 통계 쿼리라면 `fetch join`이 의도를 더 명확히 드러냈습니다.
>
> 결국 ORM이 N+1을 토해내는 순간은 "연관 로딩 전략을 다시 들여다봐야 한다"는 신호입니다.

### 변환에서 살린 점
1. 도입: 정중체 + 짧은 현실 사례 한 줄 ("센스 보강 #1").
2. 본문: bullet, 굵게 강조, 백틱 인라인 코드 — 기존 습관 그대로.
3. 비교: "두 가지 해결책"을 같은 깊이의 bullet으로 정렬.
4. 인용: "지연 로딩 + 반복 조회" 같은 핵심 표현은 따옴표로 묶음.
5. 마무리: "신호" 메타포로 한 줄 닫음 ("센스 보강 #4").
6. 어미: 시종일관 `~합니다 / ~입니다 / ~다`. 흔들리지 않음.
